Overview

This library provides a simple DSL for jsonapi.org-compliant requests. Among other things, this will help you:

  • Paginate
  • Sort
  • Filter
  • Whitelist inclusion of related resources
  • Calculate stats (totals, averages, etc)
  • Deal with customizable fieldsets
  • Guard certain behavior (based on the current user's role, for instance)
  • Deserialize incoming payloads (including nested payloads)

In other words, we provide much of the same behavior as the wonderful jsonapi-resources gem, but with a more 'traditional Rails' approach built on top of the mighty jsonapi-rb, a serialization layer similar to ActiveModelSerializers but tuned specifically to JSONAPI.

All with as little code as possible. Everything is customizable and no ActiveRecord is required (though it's a nice default).

Installation

Add to your Gemfile:

gem jsonapi_compliable

And include the mixin in your controller:

class ApplicationController < ActionController::Base
  include JsonapiCompliable
end

Quickstart - Pagination/Sorting/Sparse Fieldsets

class PeopleController < ApplicationController
  jsonapi { }

  def index
    people = People.all
    render_jsonapi(people)
  end
end

This API will now support the JSONAPI pagination and sorting parameters, in addition to sparse fieldsets. In other words, the following URLs work:

/people?page[number]=1&page[size]=20
/people?sort=name
/people?sort=-name
/people?fields[people]=name,updated_at

Filtering

Add allow_filter:

class PeopleController < ApplicationController
  jsonapi do
    allow_filter :name
  end

  # ... code ...
end

We now support /people?filter[name]=Jane. If a query comes in for a filter not allowed, JsonapiCompliable::Errors::BadFilter will be raised.

The default behavior is to assume ActiveRecord's .where(name: 'Jane'). To customize:

allow_filter :name_prefix do |scope, value|
  scope.where(["name LIKE ?", "#{value}%"])
end

Filters can be guarded:

class PeopleController < ApplicationController
  jsonapi do
    allow_filter :name, if: :admin?
  end

  def admin?
    current_user.role == :admin
  end
end

The above will raise JsonapiCompliable::Errors::BadFilter if the user is not an admin.

You can also use default_filter:

jsonapi do
  default_filter :private do |scope|
    scope.where(private: false)
  end
end

This filter can be overridden if there is a corresponding allow_filter and filter[private] is a query parameter.

Filters can be aliased. Want the URL /people?filter[name] to query full_name? Try this:

allow_filter :full_name, aliases: [:name]

Includes ("Sideloading")

jsonapi do
  includes whitelist: { index: [:siblings, {company: :address}] }
end

We'd now support the URL ?include=siblings,company.address. Any include not whitelisted will be silently dropped. If you're using ActiveRecord, everything will be automatically eager loaded.

Extra Fields

This is not part of the base jsonapi spec, but it is compatible. Imagine you have a computationally-expensive field you only want rendered when explicitly requested by the user.

class SerializablePerson < JSONAPI::Serializable::Resource
  # extra_attribute knows to only include this when requested
  extra_attribute :net_worth
end

Then add to your controller jsonapi definition:

jsonapi do
  extra_field(people: [:net_worth_in_bitcoins])
end

This field will now only be rendered when requested with /people?extra_fields=net_worth_in_bitcoins.

You can place a guard within your serializer, if needed:

class SerializablePerson < JSONAPI::Serializable::Resource
  extra_attribute :net_worth

  # This method is generated for any extra_attribute
  def allow_net_worth?
    return false unless current_user.admin?
    super
  end
end

Finally, you may need to eager load some data in order to generate this attribute. To do this, pass a block to extra_field in your controller's jsonapi definition:

extra_field(people: [:net_worth_in_bitcoins]) do |scope|
  scope.includes(:assets)
end

Stats

Add allow_stat:

class PeopleController < ApplicationController
  jsonapi do
    allow_stat total: [:count]
  end

  # ... code ...
end

We now support /people?stats[total]=count. The meta section of the response would look like:

{
  meta: {
    stats: {
      total: {
        count: 100
      }
    }
  }
}

Stats run against the normal scope, sans pagination. So if we added &filter[name_prefix]=a, we would get the total count of all people with name starting with 'a'.

The above uses a default calculation function, count, which will work out-of-the-box with ActiveRecord. Other default calculations are also supported:

allow_stat cost: [:sum, :average, :maximum, :minimum]

Each calculation can be customized (for instance, if you are not using ActiveRecord):

allow_stat :cost do
  # default behavior, equivalent to allow_stat cost: [:count]
  sum { |scope, attr| scope.count }

  # or if 'scope' is an array in this case
  sum { |scope, attr| scope.sum { |item| item.cost } }
end

Finally, arbitrary statistics can also be added:

allow_stat :cost do
  standard_deviation { |scope, attr| ... }
end

Deserialization

JSON API payloads for POST/PUT/PATCH can be a pain to deal with in Rails. Call deserialize_jsonapi! to get a Rails-friendly payload:

class PostsController < ApplicationController
  before_action :deserialize_jsonapi!, only: [:update, :create]
end

Would take this:

  {
    data: {
      type: 'posts',
      attributes: {
        title: 'my post',
      },
      relationships: {
        comments: {
          data: [
            { type: 'comments', attributes: { body: 'my comment' } }
          ]
        }
      }
    }
  }

And deserialize it to:

  {
    post: {
      title: 'my post',
      comments_attributes: [{
        body: 'my comment'
      }]
    }
  }

This differs from other deserializers in that nested relationships are converted into an accepts_nested_attributes_for-friendly payload.

You can still reference the original parameters with raw_params.

Customizing

All of the macros in the jsonapi DSL can be customized with a block. Not using ActiveRecord? No problem - you can use your ORM's scoping mechanism, or simply use the DSL to build up a hash that can be passed to an ORM of your choice:

jsonapi do
  allow_filter :name do |scope, value|
    scope[:conditions] ||= {}
    scope[:conditions].merge!(name: value)
  end

  sort do |scope, att, dir|
    scope.merge!(order: { att => dir })
  end

  paginate do |scope, current_page, per_page|
    offset = (current_page - 1 ) * per_page
    scope.merge!(limit: per_page, offset: offset)
  end

  includes whitelist: :department do |scope, includes|
    scope.merge!(include: includes)
  end
end

def index
  hash = jsonapi_scope({})
  puts hash
  # {
  #   order: { id: :asc },
  #   limit: 20,
  #   offset: 0,
  #   conditions: { name: 'foo' },
  #   include: :department
  # }
end

And here’s an example using the elasticsearch gem trample:

class EmployeesController < ApplicationController
  jsonapi do
    allow_filter :name do |scope, value|
      scope.condition(:name).eq(value)
    end

    allow_filter :name_prefix do |scope, value|
      scope.condition(:name).starts_with(value)
    end

    paginate do |scope, current_page, per_page|
      scope.metadata.pagination.current_page = current_page
      scope.metadata.pagination.per_page = per_page
      scope
    end

    sort do |scope, att, dir|
      scope.metadata.sort = [{att: att, dir: dir}]
      scope
    end

    includes whitelist: { index: :pets } do |scope, includes|
      scope.metadata.records[:includes] = includes
      scope
    end
  end

  def index
    search = jsonapi_scope(Search::Employee.new)
    search.query!

    render_jsonapi(search.records.to_a)
  end
end

render_jsonapi and jsonapi_scope

By default, render_jsonapi will automatically scope the object you pass to it. Call render_jsonapi(object, scope: false) to opt-out of this behavior.

In addition, you may want to explicitly scope before render_jsonapi to provide additional customization. The following is a typical show action:

def show
  person = jsonapi_scope(Person.all).find(params[:id])
  render_jsonapi(person)
end

With this explicit scope, all the normal scoping rules are applied (as we need include, fields, etc), then chained with find. We can then pass the record directly to render_jsonapi - it won't be scoped again.
